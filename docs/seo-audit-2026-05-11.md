# Vitalite 网站 SEO 审计报告

日期：2026-05-11  
审计对象：`kiewit-clone`，生产站点 canonical 为 `https://kkoryz.github.io/vitalite/`  
使用技能：`accio-seo-skills`、`seo-page-audit`、`seo-keyword-research`、`programmatic-seo-strategist`、`seo-competitor-analysis`、`ecommerce-seo-optimizer`、`serp-ranking-analyzer`

## 结论

Vitalite 当前不是一个普通 SPA 空壳站。它已经做了大量 SEO 基础设施：静态预渲染 fallback、canonical、hreflang、JSON-LD、sitemap、robots、`llms.txt`、长尾服务页、社区页、项目案例页和 AI/GEO 证据块。生产构建可用，TypeScript 通过。

核心风险在三处：

1. 法语路由批量质量不足：现有 SEO 预渲染测试失败，296 个法语路由低于 1000 个静态词，且大量法语 title/description 过长或混用英文。
2. 站点规模增长快：594 个可索引 URL 对一个新 GitHub Pages 子路径域名来说偏激进，应该分层 index，而不是所有 pSEO 页同时放开。
3. 竞品 SERP 的信任信号更强：Homewrap、Norseman、Grand Design Build、Plexora、ARVO 等页面普遍突出 HCRA/Tarion/BILD/WSIB、固定价格、案例数量、融资/审批/时间线等证据。Vitalite 内容深度强，但真实资质、评价、项目证据和转化信任还需要补。

## Critical Issues

### 1. 法语页面不应全部放开索引

当前有 296 个法语路由低于测试要求的 1000 静态词，很多描述超过 170-198 字符，并且标题混用英文实体，如 `Project & Gestion de construction GTA`、`Drawings, Permis & Ingenierie Toronto`。

建议：

- 短期：把低质量法语长尾和 pSEO 页 `noindex, follow`，保留 `/fr/`、`/fr/services/`、`/fr/contact-us/`、少量核心服务页可索引。
- 中期：为法语页生成真正本地化内容，而不是英文结构替换。
- 测试：把 `seo-prerender.test.mjs` 区分英文和法语阈值，或先让法语低质量页不进入 sitemap。

### 2. 全站一次性开放 594 URL 有 pSEO 质量风险

页面组合是：

- 8 个城市 x 4 个服务 = 32 location 页。
- 30 个社区 x 5 个服务 = 150 community 页。
- 23 个 long-tail guide。
- 36 个 project/case study。
- 全部双语后变成 594 个 URL。

这套结构有长尾覆盖价值，但对一个新站/子路径站点来说，应该先让最高质量、最有商业价值的 50-80 页进入 sitemap，其余先 `noindex, follow` 或不列入 sitemap。

优先保留索引：

- 首页、服务页、联系页、工具页、项目集合页。
- 8 个核心城市页。
- Toronto/North York/Markham/Richmond Hill/Vaughan/Mississauga 的 custom homes、garden suites、home additions、multiplex。
- 10-15 个最强指南：garden suite cost、laneway permit、multiplex cost、home addition permit、custom home build cost、permit-ready drawings、construction management。
- 有真实项目证据的案例页。

### 3. 信任信号弱于竞品

竞品页面常见强信号：

- Homewrap：licensed new home builder、in-house design permits、engineering-backed、insured/WSIB。
- Norseman：20+ years、architects/designers/engineers、BILD/RenoMark/Tarion 等 accreditation。
- Grand Design Build：15+ years、250+ projects、fixed price、7-year structural guarantee。
- Plexora：multiplex specialist、in-house architects/engineers、CMHC finance positioning、fourplex productized pages。
- ARVO：garden suite niche，报价工具、案例规格、价格锚点、双语。

Vitalite 页面结构比多数竞品更系统，但缺少可验证的资质、评价、许可证、真实项目照片、客户评价、团队身份、公司历史、案例编号、预算区间和施工结果证据。

## Page-Level Audit

### 首页 `/`

评分：84/100  
强项：title/description 合理，canonical/hreflang/schema 完整，静态预渲染内容充分，内部链接到服务、地点、社区、联系页。  
问题：首屏视频和大图影响 LCP；品牌资质和 proof above-the-fold 不够硬；GitHub Pages 域名弱化商业信任。  
建议：首屏加入 HCRA/Tarion/WSIB/BILD/insurance/years/project count 等真实可验证信号；压缩 logo PNG；考虑把主域接到正式品牌域名。

### 服务页 `/services/*`

评分：80/100  
强项：服务分类完整，服务页有 Service schema、FAQ、HowTo、相关链接。  
问题：部分服务 title 接近 60 字符上限，description 接近重复句式；“建筑服务、电气、permit、interior design、3D rendering”等服务的商业意图不够清晰。  
建议：每个服务页增加 1 个独有模块：适合项目、典型预算驱动因素、审批清单、交付物清单、真实项目链接。

### 地点页 `/locations/*`

评分：82/100  
强项：location context 完整，8 个城市都有定制 planning context。  
问题：4 个模板服务对 8 个城市同时索引，仍有模板相似风险。  
建议：首批只推最有需求的组合：custom home builder Toronto/North York/Markham/Richmond Hill/Vaughan/Mississauga、garden suite builder Toronto/North York/Etobicoke/Scarborough、home additions Toronto/North York/Markham、multiplex contractor Toronto/North York。

### 社区页 `/communities/*`

评分：74/100  
强项：覆盖面大，社区语境很细。  
问题：150 个英文社区服务页很容易被 Google 视为低权重相似页；没有足够真实项目照片和社区级案例支撑。  
建议：先让 20-30 个最高价值社区组合进 sitemap，其余通过 hub 可爬但 `noindex, follow`，逐步补真实 proof 后放开。

### Blog / Guide 页 `/blog/*`

评分：81/100  
强项：覆盖成本、permit、timeline、design-build vs contractor、pre-construction checklist 等强问题意图。  
问题：当前指南多数是说明性内容，缺少表格化 cost bands、permit document matrix、decision tree、downloadable checklist。  
建议：把 6 个核心指南升级为“可引用资产”：成本表、审批路径、文档清单、FAQ、官方链接、案例链接。

### Project 页 `/our-work/*`

评分：76/100  
强项：有 36 个项目页，状态、类别、地点、范围和 approval path 结构化。  
问题：部分项目 description 低于 120 字符；项目页如果图片是 AI/代表图，需要明确标注，避免信任风险；缺少真实照片、地址模糊范围、年份、面积、预算区间、permit path 证据。  
建议：项目页优先补真实“proof block”：项目类型、城市、年份、面积、scope、approval route、duration、challenge、outcome、before/after 或施工过程图。

## Programmatic SEO Strategy

当前 pSEO 方向正确，但发布策略应从“全量生成”改成“分层放量”。

推荐分层：

| Tier | 页面类型 | 处理 |
|---|---|---|
| Tier 1 | 核心服务、核心城市、核心指南、真实项目页 | `index, follow`，保留 sitemap |
| Tier 2 | 有独特本地 context 但暂无真实 proof 的城市/社区页 | `index` 但减少数量，强化内链 |
| Tier 3 | 法语长尾、弱社区组合、重复模板页 | 暂时 `noindex, follow` 或不进 sitemap |

新增 pSEO 模板建议：

- `/services/fourplex-builder-toronto/`：针对 Plexora/Modcity SERP，格式应是 productized service page，不是博客。
- `/services/laneway-suite-builder-toronto/`：与 garden suite 拆开，回应 fire access、public lane、servicing、permit。
- `/locations/{service}-{city}/projects/` 或每个 location 页加“related proof”。
- `/compare/design-build-vs-architect-toronto/`、`/compare/design-build-vs-general-contractor-gta/` 做商业比较页。

## Competitor / SERP Analysis

采样查询：

- `custom home builder Toronto design build`
- `garden suite builder Toronto permit design build`
- `multiplex contractor Toronto fourplex builder`
- `Toronto permit drawings contractor home addition`

SERP pattern：

- Custom home SERP：偏商业服务页和品牌首页，用户要看 builder credibility、项目照片、资质和 consultation CTA。
- Garden suite SERP：偏产品化服务页/报价工具/案例规格，用户要快速知道预算、时间线、permit、lot feasibility。
- Multiplex/fourplex SERP：偏 niche specialist，强调 as-of-right、unit count、financing、development charges、pro forma。
- Permit drawings SERP：偏服务页或官方信息，用户想知道需要哪些 drawings、谁能提交、多久、多少钱。

Vitalite 要赢的格式：

- 不要只写 guide。商业查询需要 service landing page。
- 每个 BOFU 页面都要有 proof：资质、项目、流程、预算驱动因素、FAQ、表格、CTA。
- 对 fourplex/multiplex 需要比泛泛“multiplex contractor”更产品化：duplex/triplex/fourplex/4+1 strategy、fire separation、egress、parking、servicing、permit path。

## Ecommerce SEO Notes

该站不是传统电商，无 SKU、价格、库存和 checkout。因此不应使用 Product schema 或虚构 rating/review/price。

但工具页和服务页可以采用“服务型转化 SEO”：

- Service schema 已适合当前业务。
- Calculator/tool 页可加 SoftwareApplication 或 WebApplication schema，前提是工具真实可用。
- 不要伪造 `aggregateRating`。
- 若以后销售固定套餐，如 feasibility review、permit drawing package，可以做 Offer schema，但必须展示真实价格/价格区间和条款。
